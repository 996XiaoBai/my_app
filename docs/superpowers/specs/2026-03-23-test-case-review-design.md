# 测试用例评审能力设计

日期：2026-03-23

## 背景

当前测试平台已经具备两条相邻但尚未闭环的能力链路：

- `review`：围绕需求文档做需求评审，输出风险和建议
- `test_case`：围绕需求上下文生成结构化测试用例，并支持导出 `Markdown`、`Excel`、`XMind`

但平台还缺少“测试用例评审”这一环。现实工作中，测试同学往往并不是缺“生成测试用例”的能力，而是缺：

- 判断已有测试用例是否真正覆盖原始需求
- 识别测试用例是否偏离需求、遗漏关键场景或引入需求外场景
- 评估测试用例是否清晰、可执行、优先级合理
- 在不覆盖原始测试用例的前提下，快速得到一份可继续加工的修订建议版

结合用户输入，本次设计聚焦于为测试平台新增 `测试用例评审` 能力，要求既能承接平台内刚生成的测试用例，也能评审外部导入的 `Markdown / Excel / XMind` 测试用例资产，但整个评审过程必须始终绑定原始需求上下文，避免评审脱离需求本身。

## 已确认需求

- 需要在测试平台增加一个新的“测试用例评审”能力
- 首版要同时支持两类入口：
  - 平台内生成后的测试用例一键送审
  - 外部导入测试用例后独立评审
- 评审时不能脱离原本需求
- 评审结果不能只给问题清单，还要能判断测试用例质量
- 外部导入测试用例时，如果没有需求上下文，不允许开始评审
- 页面入口采用双入口方案：
  - 独立新增“测试用例评审”模块
  - 在“测试用例”页面增加“一键送审”
- 评审完成后输出两类结果：
  - 评审报告
  - 修订建议版测试用例
- 首版外部导入格式需要支持：
  - 文本 / Markdown
  - Excel
  - XMind
- 首版前端不做多角色切换，但内部结构需要为未来多角色扩展留口子

## 目标与非目标

## 目标

- 新增独立的 `test_case_review` 能力模式
- 支持平台内测试用例与外部测试用例导入后的统一评审
- 严格绑定需求上下文进行评审
- 先将所有测试用例统一归一成稳定的结构化模型
- 输出结构化评审结果，支持前端分视图展示
- 输出修订建议版测试用例，并复用现有测试用例展示与导出链路
- 为后续多角色评审、整改闭环、历史对比预留扩展空间

## 非目标

- 第一阶段不支持“无需求上下文”的测试用例评审
- 第一阶段不直接覆盖原始测试用例
- 第一阶段不做前端多角色评审面板
- 第一阶段不做逐条在线编辑原始测试用例
- 第一阶段不做复杂的整改工单流转或批量整改工作流
- 第一阶段不承诺兼容所有任意结构的 Excel 和 XMind，只优先兼容平台导出格式与常见测试用例结构

## 方案选型

本次能力建设有三个备选方向。

### 方案 A：轻量拼接型

- 复用通用模块页
- 后端只输出 Markdown 评审报告
- 不专门设计结构化评审结果

优点：

- 开发快
- 改动小

缺点：

- 无法稳定承接“逐条质量判定”
- 无法自然展示“修订建议版测试用例”
- 很快会因为视图与结果结构不足而返工

### 方案 B：结构化评审型

- 新增独立 `test_case_review` 模式
- 外部导入测试用例先统一归一成 `TestCaseSuite`
- 再进入评审引擎
- 返回结构化评审结果与修订建议版用例
- 前端新增独立页面，同时在测试用例页增加“一键送审”

优点：

- 与已确认需求完全一致
- 平台内生成与外部导入可共用一套评审引擎
- 易于复用现有测试用例结果展示和导出能力
- 后续易扩展多角色评审与历史对比

缺点：

- 初始实现复杂度高于纯 Markdown 报告方案

### 方案 C：统一评审中心型

- 把需求评审和测试用例评审一起并入一个大评审中心
- 用统一模型处理多类评审对象

优点：

- 长期扩展空间大

缺点：

- 一期过重
- 导航、状态、历史、结果视图都需要较大改造
- 不适合当前范围

## 结论

采用 **方案 B：结构化评审型**。

## 设计原则

### 1. 评审必须绑定需求上下文

测试用例评审不是脱离需求的纯格式校验。平台必须将“原始需求上下文”视为一等输入，没有需求上下文时，外部导入测试用例不能进入正式评审。

### 2. 主资产不是导入文件，而是统一的 `TestCaseSuite`

无论测试用例来自平台内部生成结果、Markdown、Excel 还是 XMind，进入评审前都要先归一成同一套结构化模型。只有这样，评审、展示、导出、历史沉淀才不会各自维护一套格式。

### 3. 原始用例保留，修订建议独立输出

首版不直接覆盖原始测试用例。评审结果必须允许用户同时看到：

- 原始评审问题
- 逐条质量判定
- 修订建议版测试用例

这样才能保证结论可追溯，也便于人工二次加工。

### 4. 前端交互复用现有工作台范式

输入区优先复用 `ReviewPage` 的需求输入方式，结果区优先复用 `TestCasePage` 的结构化展示方式，避免单独发明第三套工作台交互。

### 5. 首版界面克制，结构预留扩展位

首版前端不暴露多角色切换，但后端结果模型和服务边界要支持未来从“单测试专家评审”平滑扩展到“多角色联合评审”。

## 总体架构

整体采用“五段式评审架构”：

```text
双入口
  -> 需求上下文获取
  -> 测试用例归一化
  -> 测试用例评审引擎
  -> 结构化结果与修订建议输出
```

### 1. 入口层

提供两条进入方式：

- 独立模块入口：新增“测试用例评审”
- 页面内入口：在“测试用例”结果页增加“一键送审”

### 2. 需求上下文层

复用现有需求上下文准备能力，支持：

- 手动粘贴需求
- 上传需求文档
- 读取 TAPD
- 复用最近一次上下文 `context_id`

### 3. 测试用例归一化层

将不同来源的测试用例统一转换为 `TestCaseSuite`：

- 平台内生成结果：直接读取结构化结果
- `Markdown / JSON`：通过现有解析器转换
- `Excel`：按列映射提取
- `XMind`：按脑图节点结构提取

### 4. 评审引擎层

首版以单一测试专家评审为主，重点检查：

- 是否脱离原始需求
- 是否遗漏关键验收点
- 是否引入需求外场景
- 是否存在步骤不清晰或不可执行的问题
- 用例优先级是否合理

### 5. 输出层

统一输出结构化评审结果，供：

- 页面分视图展示
- Markdown 下载与复制
- 修订建议版测试用例展示
- 历史沉淀

## 页面与交互设计

## 一、入口设计

### 独立入口

在主导航中新增“测试用例评审”模块，位置建议放在“测试工程”分组中，与“测试用例”相邻。

建议导航结构调整为：

- 测试用例
- 测试用例评审
- 测试方案
- 测试数据准备

### 一键送审入口

在 `TestCasePage` 结果工具栏中，和 `MD / TAPD / XMind` 并列新增 `送审` 按钮。

点击后执行：

- 读取当前测试用例结果 `TestCaseSuite`
- 读取当前页面的需求内容或最近上下文 `context_id`
- 将二者写入“测试用例评审”模块会话
- 跳转到测试用例评审页

如果当前没有可用需求内容且没有上下文，则按钮置灰。

## 二、测试用例评审页

页面整体沿用现有工作台布局，输入区拆成两个明确区域。

### 1. 需求上下文区

复用现有 `ReviewPage` 的输入方式：

- 粘贴需求
- 上传文档
- 读取 TAPD
- 复用最近上下文

### 2. 测试用例导入区

新增独立测试用例输入区域，支持：

- 直接带入平台内生成结果
- 粘贴 Markdown / JSON
- 上传 Excel
- 上传 XMind

如果是外部导入，页面必须先完成需求上下文输入，才能启用“开始评审”按钮。

### 3. 评审设置区

首版不暴露复杂角色选择，仅保留最必要的轻量设置，例如：

- 评审严格度
- 是否检查需求外场景

内部实现仍允许将来扩展为多角色。

## 三、结果展示

结果区建议分为五个标签页：

- `总览`
- `问题清单`
- `修订建议版用例`
- `执行日志`
- `历史记录`

### 总览

展示摘要指标：

- 高风险数量
- 中风险数量
- 低风险数量
- 需求偏离问题数量
- 覆盖不足问题数量
- 需求外场景数量

### 问题清单

按风险等级和分类展示结构化问题。每条问题至少包含：

- 风险等级
- 问题分类
- 对应需求点
- 关联测试用例
- 问题说明
- 修改建议

### 修订建议版用例

直接复用现有测试用例结果视图：

- 表格视图
- Markdown 视图

这样用户可以继续导出 `Markdown / Excel / XMind`，并复用现有操作习惯。

### 执行日志

复用现有工作台中的时间线和事件流组件。

### 历史记录

复用 `HistoryReportPanel`，按新的类型过滤历史记录。

## 后端设计

## 一、模式与调度入口

建议新增独立模式：

```python
SkillMode.TEST_CASE_REVIEW = "test_case_review"
```

需要同步改动的位置包括：

- `test_platform/core/skill_modes.py`
- `test_platform/core/services/progress_event_service.py`
- `test_platform/api/runtime.py`
- `test_platform/api/routers/run.py`
- `test_platform/core/services/review_service.py`

在 `ReviewService._execute_mode(...)` 中新增独立分支，不将该能力硬塞到现有 `review` 或 `test_case` 分支中。

## 二、服务拆分建议

建议新增专项服务，避免把导入解析和评审逻辑继续堆入 `review_service.py`：

- `TestCaseAssetParser`
  - 负责解析测试用例输入
  - 输出统一的 `TestCaseSuite`

- `TestCaseReviewService`
  - 负责基于需求上下文和 `TestCaseSuite` 执行评审
  - 输出结构化评审结果

如果一期希望减少文件数量，也可先做一个 `test_case_review_service.py`，内部再拆私有方法，但边界仍建议按“解析”和“评审”分层。

## 三、测试用例归一化设计

### 1. 平台内生成结果

直接接收现有 `TestCaseSuite` JSON，无需再次解析文本。

### 2. Markdown / JSON

优先复用现有 `parse_test_cases_from_text(...)` 解析能力，将结果再映射回 `TestCaseSuite`。

### 3. Excel

建议新增明确的列映射解析，不依赖“先转文本再让模型猜”。首版重点兼容：

- 用例名称
- 所属模块
- 前置条件
- 步骤描述
- 预期结果
- 备注
- 用例等级

当列头无法识别时，应返回明确错误，而不是静默降级。

### 4. XMind

首版优先兼容：

- 平台自身导出的 XMind 结构
- 常见的“模块 -> 用例 -> 步骤 / 预期 / 前置条件 / 优先级”树状结构

不承诺兼容任意自由排布的脑图内容。

## 四、评审引擎设计

首版采用单一测试专家评审，但要把评审维度固定下来，避免结果漂移过大。

建议固定评审维度：

- 需求一致性
- 覆盖完整性
- 需求外场景
- 步骤清晰度
- 可执行性
- 优先级合理性

其中“需求一致性”要显式检查三类问题：

- 测试用例遗漏需求中的关键验收点
- 测试用例误解需求规则
- 测试用例扩展出未被需求支持的场景

## 五、进度阶段设计

建议为 `test_case_review` 定义独立阶段：

- `context`
- `parsing_cases`
- `aligning`
- `reviewing`
- `revising`
- `done`

对应前端文案建议：

- 解析需求
- 解析用例
- 对齐需求
- 评审问题
- 生成修订版
- 已完成

## 结果契约

首版不建议只返回 Markdown，应返回结构化结果并同时附带 Markdown。

建议结构如下：

```json
{
  "summary": "已评审 26 条测试用例，识别 7 项问题，其中高风险 2 项。",
  "findings": [
    {
      "risk_level": "H",
      "category": "需求偏离",
      "related_case_ids": ["case_login_003"],
      "related_requirement_points": ["登录失败超过 5 次锁定 30 分钟"],
      "description": "当前用例只校验失败提示，未覆盖账号锁定时长与恢复条件。",
      "suggestion": "补充失败累计、锁定中登录、30 分钟后恢复登录等场景。"
    }
  ],
  "reviewed_cases": [
    {
      "case_id": "case_login_001",
      "title": "手机号验证码登录成功",
      "module": "登录",
      "verdict": "pass",
      "consistency": "aligned",
      "issues": [],
      "suggestions": []
    }
  ],
  "revised_suite": {
    "items": [],
    "summary": "修订建议版测试用例，共 29 条。",
    "markdown": "# 修订建议版测试用例"
  },
  "markdown": "# 测试用例评审报告"
}
```

### 字段说明

- `summary`
  - 供总览卡片和历史列表摘要直接使用

- `findings`
  - 供问题清单与风险总览使用

- `reviewed_cases`
  - 供后续逐条质量判定和更细粒度视图使用

- `revised_suite`
  - 直接复用现有测试用例表格、Markdown、导出链路

- `markdown`
  - 作为下载、复制、历史回放和格式兜底使用

- `ReviewPage` 的输入区工作流

## 错误处理与降级策略

### 1. 缺少需求上下文

- 外部导入场景：前端直接禁止提交
- 平台内一键送审场景：若当前无需求文本且无 `context_id`，按钮置灰

### 2. 测试用例解析失败

- 返回明确错误，例如：
  - `Excel 列头无法识别`
  - `XMind 节点结构不符合支持格式`
  - `未识别到有效测试用例`

### 3. 需求与用例映射不足

允许继续评审，但结果中要显式提示：

- 哪些需求点未能建立映射
- 哪些测试用例无法对应需求点

### 4. 结构化结果生成失败

若结构化结果构建失败，应至少返回 Markdown 报告，避免前端完全空白。

本次实现建议严格按 TDD 推进。

重点覆盖：

- `Markdown / JSON` 转 `TestCaseSuite`
- `Excel` 转 `TestCaseSuite`
- `XMind` 转 `TestCaseSuite`

重点覆盖：

- 有需求上下文时可完成评审
- 无需求上下文时拒绝外部导入评审

- `revised_suite` 能正常生成并符合 `TestCaseSuite` 契约

## 三、API 测试

重点覆盖：

- `/run`
- `/run/stream`

验证点包括：

- `test_case_review` 正常返回结构化结果
- 进度阶段映射正确
- 历史记录正确保存为 `test_case_review`

## 四、前端测试

重点覆盖：

- `parseTestCaseReviewPayload(...)`
- 测试用例页“送审”按钮写入目标会话
- 缺少需求上下文时按钮禁用
- 修订建议版用例可切换表格 / Markdown
- 历史记录过滤正确

## 分期建议

### 第一期

- 打通独立入口
- 打通平台内一键送审
- 支持 `Markdown / Excel / XMind`
- 输出结构化结果和修订建议版用例

### 第二期

- 增加多角色联合评审
- 增加问题筛选与分组能力

## 结论

本次“测试用例评审”能力建设，不应做成一个只返回 Markdown 的轻量附属功能，而应作为测试平台中“需求评审 -> 测试设计 -> 测试用例评审 -> 输出沉淀”的正式闭环节点建设。

最关键的设计决定有三个：

- 新增独立模式 `test_case_review`
- 所有输入测试用例先统一归一成 `TestCaseSuite`
- 结果同时返回 `findings / reviewed_cases / revised_suite / markdown`

这样既能满足当前明确需求，也能为后续多角色评审、整改闭环和历史对比留下稳定的扩展基础。
