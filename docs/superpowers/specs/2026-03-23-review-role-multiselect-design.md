# 评审角色多选下拉优化设计

日期：2026-03-23

## 背景

当前需求评审页中的“评审角色”采用按钮平铺方式展示。这个实现存在三个明显问题：

- 角色一多时占用横向和纵向空间，信息密度偏低
- 多选关系不够直观，用户需要依赖按钮高亮去猜测当前选中状态
- 后续角色继续增加时，页面可扩展性会快速变差

结合当前页面实现，角色选择实际属于“设置项”，而不是主操作区的核心视觉元素。用户真正需要的是：

- 快速知道当前选中了哪些评审角色
- 方便勾选或取消多个角色
- 在不打断主流程的前提下完成角色调整
- 在角色为空时得到明确反馈，而不是静默失败

因此，本次优化目标不是仅替换一个控件外观，而是把角色选择从“平铺按钮组”重构成更符合多选语义的交互模型。

## 已确认需求

- 仅优化需求评审页中的“评审角色”选择区域
- 角色控件改为多选下拉
- 下拉项支持 checkbox 多选
- 已选项在触发器中以标签压缩展示，过多时显示 `+N`
- 默认仅选中“资深测试工程师”
- 支持“清空”和“重置默认”
- 允许清空到 0 项
- 当角色为空时，点击执行评审必须拦截，并提示用户至少选择一个角色
- 不新增“资源测试工程师”角色，不新增角色类型
- 不改后端接口契约，仍然向后端传递当前已选角色数组

## 目标与非目标

### 目标

- 提升角色设置区域的信息密度
- 明确多选语义，降低误解成本
- 将默认角色调整为更符合当前使用习惯的单角色起点
- 为后续角色扩展保留稳定布局
- 保持现有评审执行链路、推荐链路、会话存储链路不变

### 非目标

- 本次不引入第三方下拉组件库
- 本次不把组件抽象成全站通用控件
- 本次不调整评审深度的交互方式
- 本次不修改后端推荐角色算法
- 本次不修改角色集合本身及角色 ID

## 现状分析

当前角色相关逻辑集中在 `ReviewPage.tsx`：

- 角色常量 `ROLES` 定义在页面文件顶部
- 默认值在会话兜底逻辑中写死为 `['test', 'product']`
- 当前切换逻辑是 `toggleRole(id)`
- 页面摘要区通过 `roleCompactLabel` 仅展示“X 角色”
- 角色区域 UI 仍然是可多选按钮组

现有仓库中没有通用 `Select`、`Popover`、`Combobox`、`MultiSelect` 组件。也就是说，如果直接抽通用控件，首版需要同时定义：

- 触发器
- 展开层
- 选中态样式
- 点击外部关闭逻辑
- 键盘和焦点行为边界

这会扩大本次范围，不利于快速稳定交付。

## 方案对比

### 方案 A：继续使用按钮组，仅压缩样式

做法：

- 缩小按钮尺寸
- 控制换行
- 通过更紧凑的样式缓解占位问题

优点：

- 改动最小
- 几乎没有交互风险

缺点：

- 没有真正表达“多选下拉”的语义
- 角色变多后仍然会失控
- 无法自然承接清空、重置默认等操作

结论：

不满足本次目标，不采用。

### 方案 B：页面内轻量多选下拉

做法：

- 仅在 `ReviewPage.tsx` 中实现一个轻量多选下拉
- 保持角色数据结构和会话存储方式不变
- 将展示摘要和默认值逻辑抽成可测试纯函数

优点：

- 改动范围小
- 风险可控
- 无需新依赖
- 能完整满足本次交互要求

缺点：

- 首版不具备跨页面复用能力

结论：

推荐采用。

### 方案 C：先抽象通用 MultiSelect 组件

做法：

- 在 `src/components/ui` 下新增通用多选组件
- 页面通过 props 传入选项、默认值、动作区内容

优点：

- 后续其他页面可复用
- 组件职责清晰

缺点：

- 当前仓库没有现成下拉基础设施
- 首版会同时引入更多样式、状态和交互边界
- 对本次单页需求来说，投入偏大

结论：

作为后续演进方向保留，本次不优先采用。

## 设计结论

采用 **方案 B：页面内轻量多选下拉**。

核心原因：

- 这次是单页面、单场景优化
- 现有代码没有通用下拉基建
- 需求更关注交互语义和默认行为，而不是组件平台化
- 先把角色选择体验做对，比提前做通用封装更重要

## 交互设计

### 1. 触发器形态

在“评审设置”中的角色区域，使用一个单独的多选触发器替代平铺按钮组。

触发器默认展示：

- 标题：`评审角色`
- 内容：已选角色摘要
- 右侧图标：展开/收起指示

### 2. 触发器摘要规则

- 0 项：显示 `请选择评审角色`
- 1 项：显示 1 个角色短标签
- 2 项：显示 2 个角色短标签
- 3 项及以上：显示前 2 个角色短标签，再显示 `+N`

这里的 `N` 表示除前 2 个以外剩余的角色数量。

为避免触发器再次被长文案撑开，触发器摘要使用短标签，示例：

- `资深测试工程师` -> `测试`
- `产品经理` -> `产品`
- `技术负责人` -> `技术`
- `设计视角` -> `设计`
- `安全专家` -> `安全`
- `资深架构师（仲裁）` -> `架构`

下拉列表中仍然显示完整角色名称，避免语义损失。

### 3. 下拉面板内容

下拉展开后包含两部分：

- 角色列表
- 底部动作区

角色列表中每个选项包含：

- checkbox
- 角色名称
- 选中态高亮

用户点击整行或 checkbox 都可以切换勾选状态。

### 3.1 展开与关闭规则

- 点击触发器：展开或收起下拉
- 点击下拉项：切换勾选，不立即关闭下拉
- 点击“清空”或“重置默认”：执行动作后保持下拉可见，方便继续确认
- 点击触发器外部区域：关闭下拉
- 按 `Escape`：关闭下拉

### 4. 默认行为

- 首次进入页面，默认角色改为 `['test']`
- 即默认只选“资深测试工程师”
- “重置默认”会把当前已选角色恢复到 `['test']`

### 5. 清空行为

- “清空”会把当前角色数组置为空
- 清空后允许界面保持 0 项状态
- 此时触发器摘要显示 `请选择评审角色`

### 6. 执行拦截

当用户点击“开始评审”且当前角色为空时：

- 不发起后端请求
- 页面显示错误提示：`请至少选择一个评审角色`
- 自动保持或展开设置区，确保用户能立刻看到修正入口

### 7. 智能推荐行为

保留现有“智能推荐”按钮。

推荐返回后：

- 用推荐角色数组直接覆盖当前选择
- 触发器摘要同步更新
- 不额外引入推荐态和手动态的区分逻辑

### 8. 特殊角色处理

`architect` 继续作为普通可选角色参与多选，不在本次增加强制单选、冲突限制或仲裁优先级规则。

### 9. 可访问性要求

- 触发器保留清晰的展开状态反馈
- 选项使用 checkbox 语义，避免仅依赖颜色表达选中态
- 焦点可见，不因自定义样式丢失聚焦提示
- 错误提示文案可直接被用户感知，不依赖悬浮态

## 状态与数据流设计

本次不修改后端接口，仅调整前端展示和前端校验。

### 1. 保持不变的数据

- `ROLES` 中的角色 ID 与标签
- `apiClient.runSkillStream(..., selectedRoles, ...)` 的调用方式
- 会话中的 `options.selectedRoles`
- `handleRecommend` 返回推荐角色并写入会话的方式

### 2. 需要调整的数据行为

- 会话默认值从 `['test', 'product']` 改为 `['test']`
- `selectedRoles` 的运行时兜底值从 `['test', 'product']` 改为 `['test']`
- 摘要展示不再只输出 `X 角色`，而是输出具体角色标签摘要

### 3. 建议抽离的纯函数

建议新增独立逻辑文件，例如：

- `src/lib/reviewRoleSelect.ts`

建议承载以下逻辑：

- 默认角色常量
- 角色短标签映射与摘要计算
- 清空与重置默认的辅助函数
- 空选择校验函数

这样做的价值是：

- 页面 JSX 保持可读
- 行为可以先写测试再实现
- 后续如果其他模块也要复用角色摘要规则，可平滑迁移

## 组件边界设计

本次建议不先抽全局通用组件，而是在 `ReviewPage.tsx` 内实现一个局部的角色多选结构。

可接受的实现方式有两种：

- 直接在 `ReviewPage.tsx` 内加入轻量下拉面板结构
- 或新增一个仅服务评审页的局部组件，例如 `ReviewRoleMultiSelect`

首选判断标准不是抽象层级，而是：

- 改动要尽量局部
- 行为要容易测试
- 不引入额外 UI 基建负担

如果 `ReviewPage.tsx` 继续膨胀明显，再把局部结构拆成页面私有子组件。

## 错误处理

本次主要增加一个前端可预期错误场景：

- 角色为空时执行评审

处理原则：

- 优先在前端阻断
- 错误文案明确、可执行
- 让用户立即看到修正位置

其余错误处理保持现状：

- 后端失败仍走原有 `error` 展示
- 推荐失败仍走原有恢复逻辑

## 风险与控制

### 风险 1：点击外部区域关闭下拉的交互细节不稳定

控制方式：

- 首版采用最小可用实现
- 只覆盖点击展开、点击选项、再次点击关闭、点击动作区等主链路

### 风险 2：页面文件继续膨胀

控制方式：

- 将摘要、默认值、校验等逻辑抽到 `src/lib/reviewRoleSelect.ts`
- JSX 中只保留展示和事件接线

### 风险 3：用户清空后不理解为什么不能执行

控制方式：

- 触发器占位文案直接提示“请选择评审角色”
- 执行时给出明确错误文案
- 自动展开设置区，降低修正成本

## 验收标准

- 评审角色区域由按钮平铺改为多选下拉
- 下拉中每个角色支持 checkbox 多选
- 默认仅选中“资深测试工程师”
- 支持清空与重置默认
- 已选项在触发器中按规则压缩展示
- 清空后允许保持 0 项
- 0 项时点击评审会被拦截并给出清晰提示
- 不影响现有推荐角色和执行评审链路
- 前端测试通过，构建通过

## 实施顺序建议

1. 先新增 `reviewRoleSelect` 纯函数测试，确认摘要、默认值、校验规则
2. 实现纯函数逻辑并跑到绿色
3. 调整 `ReviewPage` 默认角色与执行前校验
4. 将角色按钮组替换为多选下拉
5. 验证智能推荐、清空、重置默认、执行拦截与构建结果
