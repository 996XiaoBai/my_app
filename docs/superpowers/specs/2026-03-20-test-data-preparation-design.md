# 测试数据准备能力设计

日期：2026-03-20

## 背景

当前测试平台已经存在 `test_data` 能力雏形，但它本质上仍是“根据描述自由生成测试数据”的通用 AI 模式，尚未形成面向研发技术文档的专项能力。

本次设计目标是直接升级现有 `test_data` 模块，使其能够根据用户上传的研发技术文档，提取表结构信息并生成可直接使用的 MySQL SQL 语句，服务测试同学的数据准备工作。

这次能力建设不涉及真实落库，不执行数据库写入，只生成 SQL 文本结果供人工复制和使用。

## 已确认需求

- 产品入口直接复用现有 `test_data` 模块，不新增独立模块
- 前端展示名称调整为“测试数据准备”
- 后端 mode 保持 `test_data`，避免接口和历史记录映射大改
- 第一版数据库方言固定为 MySQL
- 第一版结果重点生成：
  - `SELECT`
  - `INSERT`
- 输出同时支持两种组织方式：
  - 按表输出
  - 按业务场景输出
- 输入来源是研发技术文档，重点支持技术方案、表结构说明、Confluence 导出文档等
- 对这类文档中的 HTML、编码噪音、邮件头、图片噪音需要具备清洗能力
- 第一版不做真实落库，也不做数据库连通性校验

## 目标与非目标

## 目标

- 从研发技术文档中识别表结构信息
- 输出可复制的 MySQL `SELECT` 与 `INSERT` SQL
- 同时给出按表和按场景两层结果
- 对识别不确定的内容进行显式告警
- 对结果层级做成稳定契约，便于前端展示和自动化测试

## 非目标

- 不直接连接业务数据库执行 SQL
- 不生成 `UPDATE`、`DELETE`、回滚 SQL
- 不支持多数据库方言切换
- 不尝试覆盖所有杂乱文档格式，只对常见技术文档提供鲁棒处理

## 方案选型

本次采用“文档清洗 + 结构化抽取 + SQL 生成”的混合方案。

### 备选方案对比

**方案 A：规则抽取型**

- 通过规则识别表名、字段和类型后直接生成 SQL
- 优点是稳定、快
- 缺点是场景化 SQL 能力弱

**方案 B：混合生成型**

- 先将文档抽取为统一 `schema` 中间结构
- 再基于 `schema` 生成按表 SQL 与按场景 SQL
- 优点是兼顾稳定性与表达能力，适合当前需求

**方案 C：纯 AI 自由生成型**

- 直接让模型阅读全文并自由生成 SQL
- 优点是开发快
- 缺点是结果漂移大，不适合做平台能力

### 结论

采用 **方案 B：混合生成型**。

## 模块定位

本次不新增 mode，而是升级现有 `test_data` 处理链路。

### 保留项

- `SkillMode.TEST_DATA = "test_data"`
- 前后端 mode 映射
- 历史记录类型 `test_data`
- 通用上传与执行入口

### 调整项

- 导航名称从“测试数据”改为“测试数据准备”
- 模块文案从“根据描述构造测试数据”改为“根据技术文档提取表结构并生成 MySQL SQL”
- 后端不再走全文自由生成逻辑，而是切换到专项结构化流水线

## 整体处理流程

```text
上传技术文档
  -> 文档读取
  -> 文档清洗
  -> 表结构抽取
  -> 按表 SQL 生成
  -> 按场景 SQL 生成
  -> 结果格式化
  -> 前端展示
```

### 1. 文档读取

复用现有 `document_reader`，支持常见格式：

- `.doc`
- `.docx`
- `.pdf`
- `.txt`
- `.md`
- `.html`

### 2. 文档清洗

新增面向技术文档的清洗步骤，重点处理：

- `quoted-printable` 编码解码
- Confluence 导出邮件头剥离
- HTML 标签去噪
- 样式、脚本、图片 `base64` 噪音清理
- 表格和代码块保留
- 建表片段、字段列表、字段说明归并

### 3. 表结构抽取

从清洗后的内容中提取结构化 `schema`。建议的中间模型包括：

- 表名
- 表说明
- 字段名
- 字段类型
- 字段说明
- 是否主键
- 是否必填
- 默认值
- 约束信息
- 抽取置信度

### 4. 按表 SQL 生成

基于 `schema` 为每张表生成：

- `SELECT` 模板
- `INSERT` 模板

规则：

- `SELECT` 默认列出核心字段，带基础 `WHERE` 占位
- `INSERT` 输出字段清单和值占位
- 类型不明确时不强猜具体值，只生成通用占位模板

### 5. 按场景 SQL 生成

基于表名、字段语义、文档描述生成更贴近测试动作的场景 SQL。

例如：

- 查询直播商品
- 插入直播记录
- 查询预约状态
- 插入活动配置

生成原则：

- 场景名必须可读，不用抽象编号命名
- 每个场景明确依赖哪些表
- 优先生成单表场景，能稳定识别时再生成多表场景
- 信息不足时允许降级，不强造复杂联表 SQL

### 6. 结果格式化

内部可以先维护结构化数据对象，但最终第一版仍返回 Markdown，方便复用现有前端渲染链路。

## 结果契约

第一版结果采用固定 Markdown 层级，作为前后端约定和测试契约。

```md
# 识别摘要

## 表清单
## 场景清单

# 按表 SQL

## <表名1>
### 字段摘要
### SELECT
### INSERT

## <表名2>
### 字段摘要
### SELECT
### INSERT

# 按场景 SQL

## <场景名1>
### 依赖表
### SELECT
### INSERT

## <场景名2>
### 依赖表
### SELECT
### INSERT

# 识别告警
```

### 契约要求

- 结果必须同时包含四个一级标题：
  - `识别摘要`
  - `按表 SQL`
  - `按场景 SQL`
  - `识别告警`
- 只要识别出表，`按表 SQL` 中每张表必须包含：
  - `字段摘要`
  - `SELECT`
  - `INSERT`
- 只要生成场景，`按场景 SQL` 中每个场景必须包含：
  - `依赖表`
  - `SELECT`
  - `INSERT`
- 结果中的 SQL 必须放在代码块中，便于复制

## 后端设计

建议在 `test_data` 模式下引入专项流水线，而不是继续复用全文自由生成路径。

### 推荐模块划分

**`document_sql_cleaner`**

- 输入原始文档文本
- 输出清洗后的技术文档正文
- 不负责业务推断和 SQL 生成

**`schema_extractor`**

- 输入清洗后的文本
- 输出结构化表模型
- 聚焦表、字段、约束提取

**`test_data_sql_generator`**

- 输入结构化表模型
- 输出按表 SQL 和按场景 SQL

**`test_data_result_formatter`**

- 输入结构化结果
- 输出最终 Markdown
- 保证标题层级和代码块格式稳定

### 服务编排建议

- `review_service` 中对 `SkillMode.TEST_DATA` 增加专门分支
- 该分支走专项编排方法，例如 `_run_test_data_preparation_mode`
- 不再调用当前 `_run_fulltext_skill_mode`

## 异常与降级策略

### 文档可读但未识别出表

- 返回明确提示：未识别到有效表结构
- 在 `识别告警` 中说明可能原因
- 不生成伪造 SQL

### 字段信息不完整

- `SELECT` 仍可生成
- `INSERT` 输出字段模板和值占位
- 在对应表下标记“需人工确认”

### 类型缺失或约束不明

- 不强猜高风险字段值
- 以通用占位方式输出 SQL

### 场景识别不足

- 场景 SQL 允许为空或降级
- 但 `按表 SQL` 仍然需要输出
- `识别告警` 中说明场景生成受限

### 文档噪音过高

- 尝试继续清洗
- 如仍不满足抽取条件，在告警中提示用户提供更干净的文档版本

## 测试策略

本次能力需要明确覆盖“结果层级稳定性”。

### 1. 单元测试

- 文档清洗测试
  - 能去掉邮件头
  - 能去掉 HTML 标签和样式噪音
  - 能去掉大段 `base64` 内容
- 表结构抽取测试
  - 能识别表名
  - 能识别字段名和字段类型
  - 能识别主键和注释
- SQL 生成测试
  - `SELECT` 模板符合 MySQL 常用格式
  - `INSERT` 模板包含字段和值占位

### 2. 集成测试

- 使用真实样例文档进行整链路验证
- 至少断言：
  - 能成功完成处理
  - 能识别出目标表
  - 结果中同时存在按表和按场景两类输出

### 3. 输出契约测试

必须增加固定层级断言，验证输出始终满足：

- 一级标题存在：
  - `识别摘要`
  - `按表 SQL`
  - `按场景 SQL`
  - `识别告警`
- 每张表下存在：
  - `字段摘要`
  - `SELECT`
  - `INSERT`
- 每个场景下存在：
  - `依赖表`
  - `SELECT`
  - `INSERT`

## 验收标准

第一版验收通过需满足以下条件：

1. 模块入口已从“测试数据”升级为“测试数据准备”
2. 用户上传研发技术文档后，系统可输出 MySQL `SELECT` 与 `INSERT`
3. 输出结果同时包含“按表 SQL”和“按场景 SQL”
4. 面对 Confluence 导出类 `.doc` 样例时，系统不会因 HTML 或编码噪音直接失败
5. 结果严格遵循既定标题层级
6. 识别不确定的内容会明确写入“识别告警”

## 后续扩展方向

在本设计稳定后，可沿着以下方向继续扩展：

- 增加 `UPDATE`、`DELETE`
- 增加 PostgreSQL 等方言支持
- 支持结果导出为 `.sql`
- 支持用户选择“仅按表”“仅按场景”“全部输出”
- 支持从文档中补充生成建表 SQL

## 实施建议

本次实施建议遵循“后端先稳定、前端最小改动”的原则：

- 第一阶段先打通专项链路与结果契约
- 第二阶段补充前端专属展示体验
- 第三阶段再考虑 SQL 类型扩展和导出增强

这样可以用最小代价尽快把“技术文档转测试数据 SQL”做成平台可用能力，而不是停留在概念入口或自由问答能力上。
